## Spike 验证出的技术难点（按风险排序）

### 难点1：菜单与评论没有合法全量通道（高风险，需改方案）
- 美团/大众点评不提供面向公众的菜单、评论 API。
- 爬虫面临三重墙：登录态+设备指纹；**字体加密**（页面上的数字和部分汉字用
  自定义字体映射，抓到的是乱码，需逆向字体文件且映射定期更换）；高频封号封IP。
- 法律风险真实存在：点评方多次以《反不正当竞争法》起诉抓取其评论数据的公司并胜诉。
  比赛作品要公开展示并写明数据来源，全量爬取等于自曝。
- **应对（已写进方案）**：指标降级为三层，见下文"渗透率口径"。

### 难点2：高德单查询约900条上限（已解决）
关键词搜全市必漏采。s01 用多边形搜索：先查矩形 count，>800 则一分为四递归，
直到每格安全再翻页取完。附带断点续采（done文件），配额中断第二天接着跑。

### 难点3：配额（已缓解，需实测确认额度）
高德 2025-05 起改为**月配额制**，个人认证 key 额度有限，具体以控制台为准。
缓解：QPS 限速 0.35s/次；`all_food` 全量餐饮只采一次反复使用；额度不够时
可分多天跑或多人各申请 key 轮换；指导老师若有企业认证资源优先借用。
**下一步行动：注册key后先跑 `--bbox` 小矩形，看控制台扣减速度，反推全市需要几天。**

### 难点3b：矩形bbox溢出到邻市（已解决，--clip-gz）
广州市界不规则，矩形采集框会把落在框内、市界外的相邻行政区（东莞/深圳/佛山/中山/清远/
江门/惠州）一并采进来，约占原始数据6成。分子（鸡）量小无所谓，分母（全量餐饮）量大、
邻市还都是密集城区，浪费配额可观。s01 的 `--clip-gz` 从行政区域查询拉广州市界缓存到
`data/gz_boundary.json`，四叉树采集时跳过完全落在市界外的格子（连count都不查）。
判定保守（采样点+界点兜底，宁多采不漏采），已用13个已知点验证（广州8区在内/邻市5点在外）。

### 难点4：坐标系（已解决，最容易翻车的隐性坑）
高德返回 GCJ-02；GeoScene Online 默认天地图底图 ≈ WGS-84。不纠偏整体偏移
约 300~700 米（广州塔实测偏移：东西 553m）。s02 已实现迭代逆推，误差<1m。
若最终选高德底图，则用 `--keep-gcj02` 跳过。**入库永远存原始坐标，转换只在出图前做。**

### 难点5：名称含"鸡"≠卖鸡（已解决）
田鸡(青蛙)/鸡尾酒/鸡蛋仔/鸡精商行都会被关键词捞进来。s03 用
排除词典+餐饮类目码(05开头)双重过滤，合成数据上全部正确剔除。
正式数据上需要**人工抽检 200 条**校准词典——这是上线前必做的一步。

## 渗透率口径（三层降级，写图说时要分清）

| 层 | 指标 | 数据来源 | 覆盖 | 工作量 |
|---|---|---|---|---|
| L1 | **POI渗透率** = 招牌含鸡餐饮POI / 全量餐饮POI（分区） | 高德API，s01+s03全自动 | 全市全量 | 低 |
| L2 | **菜单渗透率** = 菜单含传统鸡肴的店占比 | 分层抽样：每区按业态分层抽30~50家，人工查点评页"推荐菜"或实地拍菜单 | 抽样推断，标注置信区间 | 中（6区×40家≈240家，3人×3天） |
| L3 | **评论关键词** | 每区典型店5~10家，每家手工摘录20~30条公开评论 | 小样本定性 | 中 |

L1 是地图主图层（热力图/分区专题图），L2 用统计学口径支撑"鸡无处不在"的结论，
L3 出词云做文化佐证。三层互补，且全部经得起数据来源审查。

## 清洗规则清单（s03 实现，正式版直接复用）

R1 双重去重（高德id + 规范化名称×50m网格）→ R2 假阳性词典剔除 →
R3 餐饮类目码过滤 → R4 名称规范化(去分店括号/全半角) → R5 坐标纠偏 →
R6 三类打标（传统粤式/现代外来/其他含鸡）。
合成数据验证：38条原始 → 31条干净，7条脏数据全部被正确规则拦截。
